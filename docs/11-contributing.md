# 🤝 Contributing

> **Projeto:** LIA — Local Intelligence for Assistance
>
> **Versão:** 1.0.0
>
> **Status:** Ativo

---

# Objetivo

Este documento descreve as diretrizes para contribuição no projeto LIA.

Seu objetivo é garantir consistência, qualidade e colaboração saudável entre todos os participantes do projeto.

Independentemente do tamanho da equipe, todos deverão seguir os mesmos padrões.

---

# Nossa Filosofia

Contribuir vai muito além de escrever código.

Contribuir é ajudar o projeto a evoluir.

Valorizamos qualidade, organização, aprendizado e respeito.

Toda contribuição deve deixar o projeto melhor do que estava antes.

---

# Antes de Começar

Antes de desenvolver qualquer funcionalidade, leia:

- Vision
- Roadmap
- Backlog
- Architecture
- Tech Stack

Compreender o produto é mais importante do que escrever código rapidamente.

---

# Fluxo de Trabalho

Todo desenvolvimento deverá seguir o fluxo oficial do projeto.

```text
Issue

↓

Discussão

↓

Branch

↓

Implementação

↓

Testes

↓

Pull Request

↓

Code Review

↓

Merge
```

---

# Branches

Utilizar a convenção definida no Backlog.

Exemplos:

```text
feature/user-registration

feature/job-search

fix/login-validation

docs/update-roadmap
```

---

# Commits

Seguimos o padrão Conventional Commits.

Exemplos:

```text
feat:

fix:

docs:

refactor:

test:

style:

chore:
```

Cada commit deve representar uma mudança clara e objetiva.

---

# Pull Requests

Todo Pull Request deverá conter:

- Objetivo
- Issue relacionada
- Alterações realizadas
- Evidências (quando aplicável)
- Checklist de validação

---

# Code Review

Durante uma revisão buscamos melhorar o projeto.

Comentários devem ser respeitosos, construtivos e fundamentados.

Criticar código nunca significa criticar pessoas.

---

# Documentação

Mudanças relevantes devem atualizar a documentação correspondente.

Sempre que necessário:

- Atualizar arquivos em `docs/`
- Registrar decisões importantes em ADRs
- Atualizar diagramas
- Revisar exemplos

---

# Qualidade

Antes de abrir um Pull Request, verificar:

- Projeto compila
- Testes executados
- Código formatado
- Lint sem erros
- Documentação atualizada

---

# Comunicação

Decisões técnicas relevantes devem ser registradas.

Evitar decisões importantes apenas em conversas privadas.

O conhecimento deve permanecer no repositório.

---

# Cultura

Queremos construir um ambiente onde seja seguro:

- Fazer perguntas
- Compartilhar ideias
- Aprender
- Errar
- Evoluir

O objetivo não é provar quem sabe mais.

É construir o melhor produto possível.

---

# Filosofia

Pessoas constroem software.

Software não constrói pessoas.

Respeito, colaboração e aprendizado sempre terão prioridade.
